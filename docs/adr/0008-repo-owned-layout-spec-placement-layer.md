# 消费方自持的 Layout Spec：把「落位」拆成正交的声明式层（修订 ADR-0002）

## 状态

Accepted（设计阶段，尚未落地实现）。修订 ADR-0002 的**落位/消费方耦合**与**策略选择来源**两处；ADR-0002 的核心（可插拔切分、Unit = 同址 (image, md) 对、检测是廉价启发式、工具不入自身仓库）不变。

## 背景

技能 `digest-pdf` 明确定位为**无人值守**批处理：探测结构 → 展示一次计划 → 跑批。但真实使用反复暴露：产出的**计划本身就是错的**——`auto` 选错切分策略、目录布局不对、且不贴合它被调用时所在的项目。工具内置布局是通用且拉丁命名的（`<stem>/qNN`、`page-NNNN`、`第N章/<section>/`），而真实落点是 `~/Gaokao/gaokao-math-wiki` 这类 Obsidian 库、`gaokaowiki` 这类 Astro 站——它们有自己的中文分类树（`真题/`、`题型/`、`教材/人教A版/`…）。默认布局**几乎永远**不是目标想要的形状。这不是调参小事，是「计划正确性」问题，成本不是主因。

诊断的根因是**两个正交概念被 CONTEXT 的 "Segmentation Strategy" 定义捆在一起**：它同时管「一份 PDF 切成什么样的 Unit（*what*）」和「Unit 落在哪些目录（*where*）」。声明式落点属于后者，与前者无关——把试卷切成题（`question`）和把这些题放进 `真题/{year}/…` 是两件事。

## 决定

1. **拆分 Segmentation 与 Placement**。Segmentation Strategy 只决定 *what*（page/outline/question，内容派生，留在工具内）；**Placement** 决定 *where* + 命名，是正交的一层。CONTEXT 里 "Segmentation Strategy" 的定义去掉「and the directories that hold them」，新增 **Placement**、**Layout Spec** 两词。
2. **Layout Spec 是消费方自持的声明式 SSOT**，落在**被调用仓库**的 `<repo>/.digest/layout.yaml`（`.digest/` 留作未来工具配置目录）。它是一张**有序规则表**，每条 = `{文件名正则(具名捕获) → 切分策略 + 目标路径模板}`：
   - `match`：对 PDF 文件名（stem）的正则，具名捕获如 `year`/`region`/`subject`。语料文件名本身就编码了这些（`2016年浙江高考数学【理】（解析版）`），**零 VLM/零额外抽取**。
   - `strategy`：本条用哪个切分策略——**策略也进 spec**，一次确认、长驻 SSOT（同时收敛用户最初的两个诉求：确认目录 + 确认策略）。spec 只是**选择**分段器，不合并其逻辑，正交性保持。
   - `path`：目标路径模板，混用捕获组与分段器产出的**结构 token**（`qno`/`page`/`section`），如 `真题/{region}/{year}/{subject}/q{qno}`。**首条匹配胜出**；模板末段 token 是 Unit 的**文件名前缀**，分段器补后缀/扩展名（`-stem`、`.png`、`.md`）。
3. **模板相对仓库根解析**——即 `.digest/` 所在目录。落点由 spec 位置 + 模板**完全确定**，产出落进被调用仓库内部；`--out` 至多是运行时覆盖。工具从 cwd 向上走**自动发现** `.digest/layout.yaml`（`--layout PATH` 覆盖）。**工具本身仍对消费方无知**——耦合活在消费方自己的 spec 里，而非工具代码里（这正是对 ADR-0002「不得耦合任一消费方」的收紧而非推翻）。
4. **确认闸门 = 关于 spec 的一次性/例外事件，不是每次跑批的闸门**，从而保住「无人值守」身份：
   - spec 已存在、所有 stem 都能解析、策略无歧义 → **无人值守**跑，只打印计划。
   - 仅在以下时**阻塞并确认**：尚无 spec（技能读仓库 + 文件名**提议**一份让用户确认，确认后由**技能写入** `.digest/layout.yaml`）；提议与已存 spec 分歧；某 stem 无规则匹配；或策略有歧义。
   - stem 无匹配 → **阻塞并协助补一条规则**，不静默回退原生布局。

## 考虑并否决

- **元数据派生树**（工具自己抽 year/subject/题型 建树）：把维度逻辑塞进通用工具，且要更重的抽取；否决——文件名已含这些，正则在 spec 里解析即可，维度归属留在消费方 SSOT。
- **元数据 + spec 并用**：最强但活动部件最多；当前文件名捕获已够，暂不引入。
- **纯闸门（无环境感知）**：每次跑批都让用户手改策略/目录；否决——治标不治本，且直接杀死无人值守身份。
- **技能每次推断、逐次传参**：LLM 读 CONTEXT.md 散文推断布局，不落盘为 SSOT，跨次不确定；否决——违背声明式/SSOT。
- **conform 作第 4 个 peer 策略**：把落位重新捆回切分，无法复用 question 切题器配自定义布局，会重复切分逻辑；否决。
- **spec 相对 --out 解析 / spec 带 base 字段 + flag 覆盖**：落点分散在两处（flag + spec），SSOT 变弱、易跑错 base；否决，选「相对仓库根」。
- **spec 藏 CONTEXT.md 结构块 / 放 docs/**：把机读配置混进人读词表或散文决策区，违背 CONTEXT.md「不含实现细节」自律；否决，选独立 `.digest/layout.yaml`。
- **策略留作 per-run flag（spec 只管落位）**：概念最纯，但策略永不进 SSOT，「确认策略」要么复发要么继续依赖当初就选错的 auto；否决。

## 后果

- **改动面**：(a) Python 工具——解析 spec、按规则跑（捕获 + 模板）、与现有分段器组合、相对仓库根解析、校验（无匹配 / 缺 token / 路径冲突）；(b) `SKILL.md`——新的开场阶段，把「无人值守 / 不得询问」的绝对措辞软化为「**仓库布局一经确认即**无人值守」；(c) 文档——本 ADR + CONTEXT 三词。
- Segmentation Strategy 各变体定义里的目录形状（`第N章/<section>/`、`<stem>/qNN`…）自此读作**原生默认落位**，仅在无 Layout Spec 时生效。
- spec 是消费方 owned：换一个仓库调用，换一份 spec，工具零改动即贴合——通用性靠「耦合外置到消费方声明」实现，而非工具内建 profile。
- 待议 follow-up（不阻塞本设计）：路径冲突（两 Unit 映射到同一路径）的具体处置；`subject` 等可选捕获缺省值（如 `2018年浙江高考数学（解析版）` 无【理/文】）的模板占位默认；spec 的 schema 版本化。
